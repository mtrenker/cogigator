COGIGATOR

A strange industrial familiar for Factorio 2.0 / Space Age — a machine-mind you manufacture inside your factory. It watches a bounded production cell, diagnoses it from real game state, and — only with your approval — drafts the build. This is the build log: how the concept was scoped, constrained by real infrastructure, and turned into two competing implementation plans.

02 / The core mechanic

Two scarcities make it a Factorio problem

Both plans converge here: the assistant is not a chat overlay. It is bounded by two resources you have to build, power, and route — which is exactly what turns "AI" into a production puzzle instead of an entitlement.

◢ Sightline

Where it can look

Bought by placing Cogigator Cores / Field Stations inside production cells. No structure, no view. The assistant is physically grounded — it sees only what you plug it into.

◈ Cognition

How hard it can think

Manufactured by a compute facility you build: racks, cooling loops, multi-fluid machines, power, waste heat. Better infrastructure unlocks larger scans, deeper diagnostics, and watches.

03 / The three-layer machine

Authority never lives in a model

The safety design both plans share: a strict separation where the in-game mod is the only thing that can touch the world, and re-validates every action at apply time.

01

Pi extension — the cockpit

Model-callable read-only tools, the approval command/UI flow, compact status widgets. Talks only to the bridge. Explicitly not a security boundary.

owns · observation & approval surface
↓   narrow versioned HTTP / SSE   ↓
02

Cogigator bridge — the gateway

A new in-cluster Deployment in the factorio namespace. Owns the single RCON connection to the internal service, assembles bounded snapshots, holds inert action intents, runs a second independent permission check, exposes metrics. Never public, never authoritative.

owns · transport, snapshots, intent storage
↓   RCON (internal ClusterIP, TCP)   ↓
03

Cogigator mod (Lua) — the authority

Sole world authority. Zone scanning, the Cognition economy simulation, deterministic diagnostics, permission state, the in-game approval ledger, and every world mutation. Re-validates every intent before it touches a surface.

owns · world state & all mutations
04 / The plan duel

Two models, two blueprints

The same brief handed to two models produced genuinely different emphases — which is the point. The comparison report scores them not to crown a winner, but to see where each one peaks.

◤ Claude Opus 4.8

Infra-first & concept-led

  • RCON-first because the real cluster makes UDP the expensive path
  • Pi is the agent loop — no re-implemented LLM
  • "Manufactured cognition" promoted to the spine of the design
  • Peaks on infra safety & concept distinctiveness
◥ GPT-5.5

Contract-first & system-complete

  • Rich entity taxonomy: Datacenter, Field Stations, multi-fluid compute
  • Staged gameplay maturation, survey-room-to-compute-plant
  • Explicit testing strategy and API contracts
  • Peaks on system completeness & testability
◆ Synthesis verdict

Use Claude's infrastructure posture and cognition-as-capability spine, with GPT's entity taxonomy, testing strategy, and staged gameplay maturation. The strongest Cogigator is the hybrid.

05 / The next loop

From plan duel to repeatable experiments

The contest does not end with two plans. The next step is to build a shared Cogigator substrate and run reusable A/B tests on top of it, documenting each test in the same public format so the mod's evolution becomes part of the artifact.

◆ Experiment doctrine

One stable substrate; many competing interpretations. Each experiment records the question, substrate changes, variants, scenarios, metrics, decision, and a short timeline summary. The first proposed contest compares Sightline + Cognition Flow against Field Station + Capacity Vector.

06 / Operating discipline

What this record deliberately omits

Every document in this repo was written to be shareable. The boundary is part of the engineering, not an afterthought.

// non-sensitive by construction

No secrets, tokens, private keys, RCON passwords, credentials, raw cluster outputs, private IPs, or sealed-secret contents appear anywhere — here or in the linked files. High-level facts (Kubernetes, ArgoCD, GitOps, an internal RCON ClusterIP service, sealed-secret patterns) are described only at the level already documented. Any future Cogigator secret follows the same rule: created locally, sealed with kubeseal, and only the sealed manifest is committed.
